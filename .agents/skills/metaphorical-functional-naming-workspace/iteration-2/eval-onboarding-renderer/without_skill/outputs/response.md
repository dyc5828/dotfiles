# Naming: Cross-Platform Onboarding Renderer

## The Problem

A system that takes a single customer onboarding flow definition and generates platform-specific implementations for web, iOS, Android, and email. The core value proposition is write-once, render-everywhere for onboarding experiences.

---

## Name Suggestions

### 1. Onboard SDK

**Reasoning:** Simple, direct, developer-friendly. "SDK" signals that this is a tool developers integrate, not a standalone product. The name immediately communicates what it does without metaphor or abstraction. Works well in conversation: "Define it in Onboard SDK and it handles the rest."

**Drawback:** Generic. Hard to trademark or distinguish. "SDK" might undersell it if the system is more than a library.

---

### 2. Prism

**Reasoning:** A prism takes a single beam of white light and splits it into a full spectrum of colors. This is exactly what the system does - takes a single definition and refracts it into multiple platform-specific outputs. The metaphor is precise: one input, many outputs, each adapted to its medium. Short, memorable, easy to say. Works as a namespace: `Prism.render()`, `prism.config.yaml`.

**Drawback:** Heavily used in tech (Shopify's Prism design system, the syntax highlighter, etc.). Would need a qualifier in some contexts.

---

### 3. Loom

**Reasoning:** A loom takes a single pattern design and weaves it into fabric. The metaphor maps well - you define the pattern once, and the loom produces the material. It evokes the idea of threads coming together into something coherent across platforms. Short, one syllable, strong as a brand. "Define the flow in Loom and it weaves the native experience."

**Drawback:** Loom is already a well-known screen recording product. Collision risk is high.

---

### 4. Stencil

**Reasoning:** A stencil is a template that produces consistent results across different surfaces. You cut the stencil once and apply it to any material - paper, wood, metal, fabric. The onboarding definition is the stencil; each platform is a different surface. The name signals reusability and consistency, which are central to the product's value. Works technically: `stencil define`, `stencil render --platform ios`.

**Drawback:** Stencil.js already exists in the web component space. Could cause confusion for web developers specifically.

---

### 5. Forge

**Reasoning:** A forge takes raw material and shapes it into different forms using molds. One design, multiple castings. It conveys transformation and production - you forge the onboarding experience for each platform from a single blueprint. Has a strong, active feel. "Forge your onboarding flow once, deploy everywhere."

**Drawback:** Extremely common in tech naming. Overused to the point of being generic.

---

### 6.渡 (Wataridori) / **Passage**

**Reasoning:** Onboarding is fundamentally about passage - guiding someone from outsider to insider. "Passage" captures both the user's journey and the idea of translation across boundaries (platforms). It's elegant, human-centered, and not overly technical. Works well in docs: "Define a Passage, and the system renders it natively on every platform."

**Drawback:** Abstract enough that it doesn't immediately signal the cross-platform compilation aspect.

---

### 7. Transplant

**Reasoning:** Transplanting is taking something living and moving it to new soil where it adapts and thrives. The onboarding flow is transplanted from an abstract definition into each platform's native environment, where it takes root using native UI components. Unusual enough to be distinctive.

**Drawback:** Medical connotations might feel odd. The metaphor implies risk or delicacy, which is the opposite of the product's promise.

---

### 8. Mold

**Reasoning:** An injection mold produces identical shapes from different materials - plastic, rubber, metal. The onboarding definition is the mold; each platform render is a different material poured into it. Precise metaphor for write-once, render-everywhere. Short. Clear.

**Drawback:** "Mold" also means fungus. Unfortunate double meaning.

---

### 9. Turnstile

**Reasoning:** A turnstile is a universal gate that everyone passes through. It controls and standardizes the entry experience regardless of who is entering. The name directly references onboarding as an entry point, and "turn" evokes the idea of translating or converting between formats. Distinctive in the tech space.

**Drawback:** Cloudflare's CAPTCHA product is called Turnstile. Also, turnstiles have a slightly negative "gatekeeper" connotation.

---

### 10. Rosetta

**Reasoning:** The Rosetta Stone contained the same message written in three different languages, enabling translation between them. This system contains the same onboarding flow rendered in multiple platform languages. The metaphor is sharp: one meaning, multiple expressions. Developers already associate "Rosetta" with translation layers (Apple's Rosetta for x86-to-ARM). "Define it once in Rosetta and it speaks every platform's language."

**Drawback:** Apple's Rosetta is well-known. Might cause initial confusion, though the domains are different enough.

---

## Top Recommendations

If I had to pick three:

1. **Prism** - Best metaphor-to-function alignment. One input, multiple platform-specific outputs. Instantly graspable.

2. **Stencil** - Best at communicating the write-once, apply-everywhere value prop. Feels like a tool, which matches the developer experience.

3. **Rosetta** - Best at communicating the cross-platform translation aspect. Carries strong existing associations with "making things work across different systems."

The strongest overall pick is **Prism**. The metaphor is precise, the name is short, and it works naturally in both marketing and technical contexts. The namespace collision issue is manageable with context or a light modifier if needed.
